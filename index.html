How it works

Four moves from drawing to dashboard.

Built the way progress is actually reported on a linear job — by route, by chainage, by activity.

0+000

Load the drawing

Drop in your layout key plan as an image or PDF. Each service layer can carry its own drawing on the same board.

0+100

Add routes

Trace a centreline in a few clicks with its start and end chainage — or auto-detect the coloured network lines and let OCR read the chainage labels for you.

0+200

Log progress

From CH, to CH, completed activity — typed or picked directly on the drawing. Carriageways split LHS / RHS / both.

0+300

Report

Stage-weighted % per route, side and layer, installed quantity vs target, annotated PNG, printable PDF report and a full CSV log.

What it does

Everything a section engineer marks up by hand, kept live.

Progress model

Stage-weighted % complete

Each layer carries its real activity sequence — Excavation → Laying → Complete — and every metre earns credit for the stages it has passed, not just "done / not done".

Detection

Auto-detect routes

Pick a line colour on the drawing and the tool extracts the network as routes, with best-effort OCR of chainage labels. Review, correct, add.

Collaboration

One shared live board

Optional Firebase backend syncs every route and entry across the team in seconds, with each update attributed by name. View-only mode for lookers.

Multi-project

Boards per project

Every URL hash is an independent board — #r2200, #sector-6 — switchable from the toolbar. One deployment serves the whole portfolio.

Quantities

Installed vs target

Set a target quantity per layer and track metres (or chamber counts) installed through the last physical activity against it.

Outputs

Deck-ready exports

Annotated PNG with a title strip (layer, %, cut-off date, legend) for the weekly deck, printable report with the full progress log, CSV for Excel.

Deploy

Your copy, your data, in ten minutes.

The tool is a single HTML file. Local mode works with zero setup; shared boards need a free Firebase Realtime Database.

  1. Fork or clone the repository and enable GitHub Pages — the landing page and tool are served as-is.
  2. For team sync, create a Firebase project, add a Realtime Database, and paste the web-app config into app.html.
  3. Apply the database rules from database.rules.json before sharing the link.
The Firebase config in the file is public by design — access control lives entirely in your database rules. Leaving the default open rules on a published page means anyone with the URL can read and overwrite your boards. Lock the rules first; the README walks through it.
